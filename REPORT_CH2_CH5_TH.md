# เนื้อหาสำหรับรายงาน
## บทที่ 2 ทฤษฎี และงานวิจัยที่เกี่ยวข้อง

### 2.1 ทฤษฎีที่เกี่ยวข้อง
การตรวจจับวัตถุ (Object Detection) เป็นงานในสาขาคอมพิวเตอร์วิทัศน์ที่ต้องระบุตำแหน่งและประเภทของวัตถุในภาพพร้อมกัน โดยผลลัพธ์มักอยู่ในรูปของกรอบสี่เหลี่ยมล้อมวัตถุ (Bounding Box) และคะแนนความเชื่อมั่น (Confidence Score) แนวคิดหลักประกอบด้วยการแทนตำแหน่งวัตถุ การคำนวณความทับซ้อนระหว่างกรอบ (Intersection over Union: IoU) และกระบวนการคัดกรองกรอบซ้ำ (Non-Maximum Suppression: NMS)

รูปแบบป้ายกำกับข้อมูลของ YOLO ใช้พิกัดแบบ normalized คือ
`<class_id> <x_center> <y_center> <width> <height>`
ซึ่งช่วยให้โมเดลเรียนรู้ได้อย่างมีประสิทธิภาพและรองรับภาพขนาดต่างกันโดยไม่ต้องเปลี่ยนรูปแบบป้ายกำกับ

### 2.2 อัลกอริทึม YOLO
YOLO (You Only Look Once) เป็นสถาปัตยกรรมตรวจจับวัตถุแบบ one-stage detector ที่ประมวลผลภาพและทำนายผลในขั้นตอนเดียว จึงมีข้อเด่นด้านความเร็วเทียบกับหลายวิธีแบบ two-stage detector โครงสร้างของ YOLOv8/Ultralytics โดยทั่วไปประกอบด้วย
- Backbone สำหรับสกัดคุณลักษณะภาพ
- Neck สำหรับรวมคุณลักษณะหลายสเกล
- Detection Head สำหรับทำนายกรอบวัตถุและคลาส

ในงานนี้ใช้โมเดลเริ่มต้น `yolov8n.pt` ซึ่งเป็นรุ่นขนาดเล็ก เน้นความเร็วและการทดลองซ้ำได้รวดเร็ว เหมาะกับการพัฒนาและปรับจูนเบื้องต้น

### 2.3 ตัวชี้วัดประสิทธิภาพ
การประเมินโมเดลใช้ตัวชี้วัดหลักดังนี้
- Precision: สัดส่วนผลทำนายที่ถูกต้องจากผลทำนายทั้งหมด
- Recall: สัดส่วนวัตถุจริงที่ตรวจพบได้
- mAP@0.5: ค่าเฉลี่ยความแม่นยำที่เกณฑ์ IoU = 0.5
- mAP@0.5:0.95: ค่าเฉลี่ยความแม่นยำหลายเกณฑ์ IoU (เข้มงวดกว่า)

ตัวชี้วัดดังกล่าวใช้ร่วมกันเพื่อสะท้อนทั้งความครบถ้วนและความแม่นยำของการตรวจจับ

### 2.4 งานวิจัยที่เกี่ยวข้อง
งานวิจัยด้าน object detection โดยเฉพาะตระกูล YOLO มีพัฒนาการต่อเนื่องจากความเร็วสูงในรุ่นแรก ไปสู่ความแม่นยำที่ดีขึ้นในรุ่นใหม่ผ่านการออกแบบโครงสร้างเครือข่าย การจัดการฟีเจอร์หลายสเกล และเทคนิคเพิ่มข้อมูล (Data Augmentation)

สำหรับงานที่มีวัตถุเฉพาะโดเมน (เช่น วัตถุทางการแพทย์ ชิ้นส่วนอุตสาหกรรม หรืออวัยวะสัตว์) มักพบว่าปัจจัยสำคัญต่อผลลัพธ์คือ
1) คุณภาพการทำ annotation
2) ความหลากหลายของสภาพแสง/มุมมอง
3) ความสมดุลของข้อมูล
4) การตั้งค่า threshold และ hyperparameters ที่เหมาะสม

ในรายงานฉบับสมบูรณ์ ควรเพิ่มเอกสารอ้างอิงเชิงวิชาการ (บทความวารสาร/ประชุมวิชาการ) อย่างน้อย 5–10 แหล่ง โดยเน้นงาน YOLO รุ่นล่าสุดและงานประยุกต์ในโดเมนใกล้เคียง

## บทที่ 3 การออกแบบและการพัฒนา

### 3.1 ภาพรวมสถาปัตยกรรมระบบ
ระบบในงานนี้ถูกออกแบบให้ทำงานเป็นลำดับขั้นแบบ pipeline ตั้งแต่การเตรียมข้อมูล การฝึกโมเดล การทดสอบ และการนำผลตรวจจับไปใช้งานต่อ โดยโครงสร้างหลักแบ่งเป็นโมดูลจัดการข้อมูล โมดูลฝึก โมดูลทดสอบ และโมดูลครอปภาพ กระบวนการทั้งหมดไหลจากข้อมูลดิบไปสู่ชุดข้อมูลมาตรฐานสำหรับการเรียนรู้ จากนั้นจึงสร้างผลการฝึกและต่อยอดเป็นผลลัพธ์เชิงใช้งานจริงในการตรวจจับและครอปภาพ

แนวคิดสำคัญของการออกแบบคือการแยกหน้าที่ของแต่ละส่วนให้ชัดเจน เพื่อลดความซับซ้อนและทำให้ตรวจสอบข้อผิดพลาดได้ง่ายขึ้น เมื่อมีการปรับพารามิเตอร์หรือเปลี่ยนรุ่นโมเดล ผู้พัฒนาสามารถแก้ไขเฉพาะจุดโดยไม่ต้องปรับทั้งระบบ อีกทั้งยังมีการใช้ไฟล์ตั้งค่ากลางเพื่อควบคุมการทดลองให้เป็นมาตรฐานเดียวกัน ทำให้การทดลองหลายรอบทำได้เป็นระบบและเปรียบเทียบผลลัพธ์ได้สะดวกมากขึ้น

### 3.2 การเตรียมข้อมูล
การเตรียมข้อมูลเริ่มจากการจัดเก็บภาพต้นฉบับและป้ายกำกับแยกตามประเภทอย่างเป็นระบบ จากนั้นใช้โมดูลเตรียมข้อมูลเพื่อสร้างโครงสร้างชุดข้อมูลตามมาตรฐานของ YOLO และแบ่งข้อมูลออกเป็นชุด train, validation และ test ตามสัดส่วนที่กำหนดไว้ กระบวนการนี้ยังตรวจสอบความสอดคล้องของชื่อไฟล์ภาพกับไฟล์ป้ายกำกับ เพื่อให้แน่ใจว่าข้อมูลแต่ละภาพมี annotation ที่ตรงกัน และรองรับนามสกุลภาพหลายชนิดเพื่อความยืดหยุ่นในการใช้งาน

ในเชิงวิธีวิทยา มีการกำหนดค่า `seed` สำหรับการสุ่มแบ่งข้อมูลเพื่อให้ผลการทดลองทำซ้ำได้ การออกแบบในลักษณะนี้ช่วยลดความคลาดเคลื่อนที่เกิดจากการแบ่งข้อมูลแบบสุ่มต่างกันในแต่ละรอบทดลอง ทั้งยังลดปัญหาข้อมูลสูญหายหรือจัดวางผิดตำแหน่งที่มักพบเมื่อจัดการไฟล์ด้วยมือ ดังนั้นคุณภาพของขั้นตอนเตรียมข้อมูลจึงเป็นฐานสำคัญที่ส่งผลโดยตรงต่อความแม่นยำของโมเดลในขั้นตอนถัดไป

### 3.3 การออกแบบการฝึกโมเดล
การออกแบบการฝึกโมเดลใช้ไฟล์ตั้งค่ากลางเป็นจุดศูนย์กลางในการกำหนดพารามิเตอร์ทั้งหมด ทั้งพาธข้อมูล คลาสเป้าหมาย พารามิเตอร์การฝึก และการทำ augmentation ทำให้การจัดการการทดลองเป็นระบบเดียวกันทุกครั้ง เมื่อเริ่มกระบวนการฝึก โปรแกรมจะดำเนินการตามลำดับที่ชัดเจน คือโหลดค่าตั้งต้น ตรวจสอบความพร้อมของข้อมูล โหลดโมเดลตั้งต้น แล้วเริ่มฝึกพร้อมบันทึกผลลัพธ์และ checkpoint โดยอัตโนมัติ

ในงานนี้เลือกใช้ `yolov8n.pt` เพื่อให้เหมาะกับการทดลองหลายรอบและลดเวลาในการฝึก ขณะที่ค่า `imgsz=640` ให้สมดุลระหว่างรายละเอียดภาพและต้นทุนการประมวลผล ส่วน `optimizer=SGD` ถูกเลือกเพื่อเสถียรภาพของการเรียนรู้ระยะยาว และการเปิดใช้ augmentation หลายรูปแบบช่วยเพิ่มความหลากหลายเชิงภาพเพื่อบรรเทาปัญหา overfitting นอกจากนี้การกำหนด `patience` สำหรับ early stopping ยังช่วยควบคุมเวลาฝึกและลดการคำนวณที่ไม่จำเป็นเมื่อประสิทธิภาพไม่ดีขึ้นอย่างมีนัยสำคัญ

### 3.4 การออกแบบการทดสอบและใช้งาน
ส่วนทดสอบและใช้งานถูกออกแบบให้รองรับทั้งภาพเดี่ยว โฟลเดอร์ภาพ วิดีโอ และเว็บแคมแบบเรียลไทม์ โดยผู้ใช้สามารถใช้รูปแบบคำสั่งเดียวกันและให้ระบบตรวจสอบชนิดแหล่งข้อมูลเพื่อเลือกวิธีประมวลผลที่เหมาะสมโดยอัตโนมัติ หลังจากโมเดลทำนายผลแล้ว โปรแกรมจะแยกข้อมูลสำคัญ ได้แก่ กล่องวัตถุ ค่าความเชื่อมั่น หมายเลขคลาส และชื่อคลาส เพื่อนำเสนอผลในรูปแบบที่อ่านง่าย พร้อมวาดกรอบและข้อความกำกับบนภาพก่อนบันทึก

การออกแบบนี้มีความยืดหยุ่นสูงในทางปฏิบัติ เพราะผู้ใช้สามารถปรับค่า `conf` เพื่อควบคุมความเข้มงวดของการทำนาย และปรับค่า `iou` เพื่อจัดการกรอบที่ซ้อนทับกันจากกระบวนการ NMS ได้ตามเงื่อนไขหน้างานจริง จึงรองรับทั้งการประเมินเชิงวิจัยและการใช้งานเชิงปฏิบัติการที่ต้องการความรวดเร็ว

### 3.5 การพัฒนาเครื่องมือครอปภาพ
เครื่องมือครอปภาพถูกพัฒนาเป็นส่วนติดต่อผู้ใช้แบบกราฟิกด้วย Tkinter เพื่อให้ผู้ใช้ที่ไม่ถนัดคำสั่งบนเทอร์มินัลสามารถใช้งานได้ง่ายขึ้น ผู้ใช้เพียงเลือกโมเดล เลือกภาพอินพุต และกำหนดโฟลเดอร์ปลายทาง จากนั้นระบบจะทำการตรวจจับวัตถุและครอปภาพย่อยจากพิกัดกรอบที่ทำนายได้โดยอัตโนมัติ ขั้นตอนดังกล่าวช่วยลดภาระงานที่เดิมต้องครอปด้วยมือ และลดความผิดพลาดจากการตัดภาพไม่ตรงตำแหน่ง

อีกจุดสำคัญของการออกแบบคือการตั้งชื่อไฟล์ผลลัพธ์ให้มีความหมาย โดยประกอบด้วยชื่อภาพเดิม เวลา ลำดับวัตถุ คลาส และค่าความเชื่อมั่น ทำให้สามารถตรวจสอบย้อนหลังและบริหารจัดการข้อมูลได้เป็นระบบ โดยเฉพาะเมื่อมีการใช้งานในปริมาณมาก โดยสรุปแล้วโมดูลครอปภาพทำหน้าที่เป็นสะพานเชื่อมจากผลการตรวจจับไปสู่การใช้งานต่อยอด เช่น การคัดเลือกตัวอย่าง การสร้างข้อมูลรอบใหม่ หรือการป้อนข้อมูลให้ระบบ downstream ในขั้นตอนถัดไป

### 4.4 ผลการทดลองเชิงภาพ
ไฟล์รูปสำคัญที่ใช้วิเคราะห์ผล ได้แก่
- `results.png` : แสดงแนวโน้มค่า loss และตัวชี้วัดหลักตลอดทุก epoch ใช้ดูว่าการฝึกมีทิศทางดีขึ้นต่อเนื่องหรือไม่ หากกราฟ loss ลดลงและค่า mAP/precision/recall สูงขึ้นอย่างสม่ำเสมอ จะสะท้อนว่าโมเดลเรียนรู้ได้มีเสถียรภาพ
- `confusion_matrix.png` : แสดงจำนวนการทำนายถูก/ผิดในรูปแบบเมทริกซ์จริง (ค่าจำนวนเต็ม) ช่วยระบุว่าคลาสใดสับสนกับคลาสใด และช่วยมองปัญหา false positive / false negative ได้ชัดเจน
- `confusion_matrix_normalized.png` : เป็น confusion matrix แบบปรับสัดส่วนให้อยู่ในรูปเปอร์เซ็นต์ ทำให้เปรียบเทียบประสิทธิภาพระหว่างคลาสได้ง่ายขึ้น แม้จำนวนตัวอย่างของแต่ละคลาสไม่เท่ากัน
- `BoxPR_curve.png` : กราฟความสัมพันธ์ระหว่าง Precision และ Recall ใช้ประเมินคุณภาพการตรวจจับโดยรวม โดยกราฟที่เข้าใกล้มุมขวาบนมากกว่าแปลว่าระบบรักษาสมดุลระหว่างความแม่นยำและความครบถ้วนได้ดี
- `BoxP_curve.png` : กราฟ Precision ตามค่า confidence threshold ใช้พิจารณาว่าเมื่อเพิ่มความเข้มงวดในการยอมรับผลทำนายแล้ว ความแม่นยำเปลี่ยนอย่างไร เหมาะสำหรับเลือก threshold เพื่อลดการแจ้งเตือนผิด
- `BoxR_curve.png` : กราฟ Recall ตามค่า confidence threshold ใช้ดูผลกระทบต่อการตรวจพบวัตถุจริงเมื่อปรับ threshold โดยทั่วไปหาก threshold สูงขึ้น Recall มักลดลง จึงใช้ประกอบการหาค่าที่เหมาะสม
- `BoxF1_curve.png` : กราฟ F1-score ตาม threshold ซึ่งสะท้อนสมดุลระหว่าง Precision และ Recall จุดสูงสุดของกราฟมักถูกใช้เป็นค่าอ้างอิงในการเลือก threshold สำหรับใช้งานจริง
- `val_batch0_labels.jpg` : ภาพตัวอย่างข้อมูลจริงในชุด validation (ground truth) ใช้ตรวจสอบคุณภาพ annotation และความสมเหตุสมผลของตำแหน่งกรอบอ้างอิง
- `val_batch0_pred.jpg` : ภาพผลทำนายของโมเดลบนชุด validation ใช้เปรียบเทียบกับภาพ labels เพื่อตรวจสอบความคลาดเคลื่อนเชิงตำแหน่ง จำนวนวัตถุที่ตรวจพบ และความเหมาะสมของ confidence

จากภาพรวมพบว่าเส้นโค้งและ confusion matrix สอดคล้องกับผลตัวเลขที่ดี แสดงถึงความเสถียรของโมเดลในชุด validation

### 4.5 อภิปรายผล
แม้ผลเชิงตัวเลขสูงมาก แต่ยังควรพิจารณาประเด็นต่อไปนี้
- ความสามารถในการทั่วไปกับข้อมูลนอกโดเมน (domain shift)
- ความไวต่อแสง มุมกล้อง และการบังบางส่วน
- โอกาสเกิด overfitting หากข้อมูลมีความคล้ายกันสูง

จึงควรมีการทดสอบภาคสนามเพิ่มเติมและเพิ่มความหลากหลายข้อมูลในอนาคต

## บทที่ 5 บทสรุปและข้อเสนอแนะ

### 5.1 บทสรุป
โครงการนี้สามารถพัฒนาระบบตรวจจับวัตถุแบบกำหนดคลาสเองได้ครบวงจร ตั้งแต่การเตรียมข้อมูล การฝึกโมเดล การทดสอบ ไปจนถึงการครอปภาพเพื่อนำไปใช้งานต่อ ผลการทดลองชี้ให้เห็นว่าโมเดลมีประสิทธิภาพสูงสำหรับคลาสเป้าหมาย และระบบที่พัฒนามีความพร้อมสำหรับการใช้งานจริงในระดับต้นแบบ

### 5.2 ข้อเสนอแนะ
1. เพิ่มจำนวนและความหลากหลายของข้อมูลฝึก เพื่อเพิ่มความทนทานของโมเดล
2. เพิ่มหลายคลาสใน `names` เพื่อรองรับงานที่ซับซ้อนขึ้น
3. ทดลองโมเดลขนาดอื่น (`yolov8s`, `yolov8m`) เพื่อเปรียบเทียบความแม่นยำกับเวลา
4. จัดทำชุดทดสอบภายนอกที่ไม่ซ้ำโดเมนฝึกเพื่อประเมินความสามารถการทั่วไป
5. พัฒนาระบบ deployment (เช่น API หรือแอป) เพื่อใช้งานในงานจริง

### 5.3 แนวทางพัฒนาต่อ
- เพิ่มระบบติดตามเวอร์ชันข้อมูลและโมเดล
- สร้าง dashboard สำหรับติดตามผลการทดลองหลายรอบ
- เพิ่มขั้นตอน post-processing เฉพาะโดเมนเพื่อเพิ่มความแม่นยำเชิงปฏิบัติ
