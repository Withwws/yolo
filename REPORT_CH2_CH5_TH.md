# เนื้อหาสำหรับรายงาน
## บทที่ 2 ทฤษฎี และงานวิจัยที่เกี่ยวข้อง

### 2.1 ทฤษฎีที่เกี่ยวข้อง
การตรวจจับวัตถุ (Object Detection) เป็นงานในสาขาคอมพิวเตอร์วิทัศน์ที่ต้องระบุตำแหน่งและประเภทของวัตถุในภาพพร้อมกัน โดยผลลัพธ์มักอยู่ในรูปของกรอบสี่เหลี่ยมล้อมวัตถุ (Bounding Box) และคะแนนความเชื่อมั่น (Confidence Score) แนวคิดหลักประกอบด้วยการแทนตำแหน่งวัตถุ การคำนวณความทับซ้อนระหว่างกรอบ (Intersection over Union: IoU) และกระบวนการคัดกรองกรอบซ้ำ (Non-Maximum Suppression: NMS)

รูปแบบป้ายกำกับข้อมูลของ YOLO ใช้พิกัดแบบ normalized คือ
`<class_id> <x_center> <y_center> <width> <height>`
ซึ่งช่วยให้โมเดลเรียนรู้ได้อย่างมีประสิทธิภาพและรองรับภาพขนาดต่างกันโดยไม่ต้องเปลี่ยนรูปแบบป้ายกำกับ

### 2.2 อัลกอริทึม YOLO
YOLO (You Only Look Once) เป็นสถาปัตยกรรมตรวจจับวัตถุแบบ one-stage detector ที่ประมวลผลภาพและทำนายผลในขั้นตอนเดียว จึงมีข้อเด่นด้านความเร็วเทียบกับหลายวิธีแบบ two-stage detector โครงสร้างของ YOLOv8/Ultralytics โดยทั่วไปประกอบด้วย
- Backbone สำหรับสกัดคุณลักษณะภาพ
- Neck สำหรับรวมคุณลักษณะหลายสเกล
- Detection Head สำหรับทำนายกรอบวัตถุและคลาส

ในงานนี้ใช้โมเดลเริ่มต้น `yolov8n.pt` ซึ่งเป็นรุ่นขนาดเล็ก เน้นความเร็วและการทดลองซ้ำได้รวดเร็ว เหมาะกับการพัฒนาและปรับจูนเบื้องต้น

### 2.3 ตัวชี้วัดประสิทธิภาพ
การประเมินโมเดลใช้ตัวชี้วัดหลักดังนี้
- Precision: สัดส่วนผลทำนายที่ถูกต้องจากผลทำนายทั้งหมด
- Recall: สัดส่วนวัตถุจริงที่ตรวจพบได้
- mAP@0.5: ค่าเฉลี่ยความแม่นยำที่เกณฑ์ IoU = 0.5
- mAP@0.5:0.95: ค่าเฉลี่ยความแม่นยำหลายเกณฑ์ IoU (เข้มงวดกว่า)

ตัวชี้วัดดังกล่าวใช้ร่วมกันเพื่อสะท้อนทั้งความครบถ้วนและความแม่นยำของการตรวจจับ

### 2.4 งานวิจัยที่เกี่ยวข้อง
งานวิจัยด้าน object detection โดยเฉพาะตระกูล YOLO มีพัฒนาการต่อเนื่องจากความเร็วสูงในรุ่นแรก ไปสู่ความแม่นยำที่ดีขึ้นในรุ่นใหม่ผ่านการออกแบบโครงสร้างเครือข่าย การจัดการฟีเจอร์หลายสเกล และเทคนิคเพิ่มข้อมูล (Data Augmentation)

สำหรับงานที่มีวัตถุเฉพาะโดเมน (เช่น วัตถุทางการแพทย์ ชิ้นส่วนอุตสาหกรรม หรืออวัยวะสัตว์) มักพบว่าปัจจัยสำคัญต่อผลลัพธ์คือ
1) คุณภาพการทำ annotation
2) ความหลากหลายของสภาพแสง/มุมมอง
3) ความสมดุลของข้อมูล
4) การตั้งค่า threshold และ hyperparameters ที่เหมาะสม

ในรายงานฉบับสมบูรณ์ ควรเพิ่มเอกสารอ้างอิงเชิงวิชาการ (บทความวารสาร/ประชุมวิชาการ) อย่างน้อย 5–10 แหล่ง โดยเน้นงาน YOLO รุ่นล่าสุดและงานประยุกต์ในโดเมนใกล้เคียง

## บทที่ 3 การออกแบบและการพัฒนา

### 3.1 ภาพรวมสถาปัตยกรรมระบบ
ระบบถูกออกแบบเป็น 4 ส่วนหลัก
1. ส่วนเตรียมข้อมูล (`prepare_data.py`)
2. ส่วนฝึกโมเดล (`train.py` + `config.yaml`)
3. ส่วนทดสอบและใช้งาน (`test.py`)
4. ส่วนครอปภาพจากผลตรวจจับ (`crop.py`)

โครงสร้างนี้ช่วยให้แยกความรับผิดชอบชัดเจน และปรับปรุงทีละส่วนได้สะดวก

### 3.2 การเตรียมข้อมูล
ข้อมูลต้นฉบับเก็บในโฟลเดอร์ `raw_data/images` และ `raw_data/labels` จากนั้นสคริปต์ `prepare_data.py` ทำการ
- สร้างโครงสร้าง dataset มาตรฐาน YOLO
- แบ่งข้อมูลเป็น train/val/test ตามสัดส่วนที่กำหนด (ค่าเริ่มต้น 0.7/0.2/0.1)
- คัดลอกภาพและ label ให้ตรงคู่กัน

แนวทางนี้ลดข้อผิดพลาดจากการจัดไฟล์ด้วยมือ และทำให้กระบวนการทดลองทำซ้ำได้

### 3.3 การออกแบบการฝึกโมเดล
ไฟล์ `config.yaml` ใช้กำหนดพารามิเตอร์สำคัญ เช่น
- ชุดข้อมูล: `path`, `train`, `val`, `test`
- คลาส: `names` (งานนี้กำหนด `0: dogNose`)
- พารามิเตอร์ฝึก: `epochs`, `batch`, `imgsz`
- ตัวปรับเหมาะสม: `optimizer`, `lr0`, `momentum`, `weight_decay`
- Augmentation: `hsv_*`, `translate`, `scale`, `fliplr`, `mosaic`

`train.py` ตรวจสอบความพร้อมของข้อมูลก่อนเริ่มฝึก เพื่อป้องกันข้อผิดพลาดตั้งแต่ต้นทาง และบันทึกผลการฝึกอัตโนมัติไว้ในโฟลเดอร์ผลลัพธ์

### 3.4 การออกแบบการทดสอบและใช้งาน
`test.py` รองรับแหล่งข้อมูลหลายรูปแบบ
- ภาพเดี่ยว
- โฟลเดอร์ภาพ
- วิดีโอ
- เว็บแคมแบบเรียลไทม์

ผู้ใช้สามารถกำหนดค่า `conf` และ `iou` เพื่อควบคุมความไวของระบบให้เหมาะกับโจทย์จริง

### 3.5 การพัฒนาเครื่องมือครอปภาพ
`crop.py` พัฒนาเป็น GUI ด้วย Tkinter ช่วยให้ผู้ใช้ทั่วไปใช้งานได้ง่ายโดยไม่ต้องพิมพ์คำสั่ง ประกอบด้วยความสามารถหลัก
- เลือกไฟล์โมเดล `.pt`
- เลือกภาพอินพุต
- กำหนดโฟลเดอร์บันทึก
- ตรวจจับและบันทึกภาพครอปอัตโนมัติ

ชื่อไฟล์ครอปถูกสร้างแบบมีเวลาและคลาสกำกับ ช่วยให้ติดตามผลและจัดการข้อมูลสะดวก

## บทที่ 4 การทดลองและผลการทดลอง

### 4.1 การตั้งค่าการทดลอง
การทดลองใช้สคริปต์ฝึกในโครงการนี้โดยอ้างอิงค่าจาก `config.yaml` และประเมินผลจากไฟล์ผลลัพธ์ที่ระบบบันทึกอัตโนมัติ
- โมเดลตั้งต้น: `yolov8n.pt`
- จำนวน epoch: 100
- image size: 640
- optimizer: SGD
- คลาสที่ตรวจจับ: dogNose

### 4.2 ชุดข้อมูลที่ใช้
ข้อมูลภาพและ label ถูกจัดเก็บตามมาตรฐาน YOLO และแบ่งเป็น train/val/test ผ่าน `prepare_data.py` โดยมีตัวอย่างชื่อภาพในชุดข้อมูล เช่น
- `01ba27f3-20251217_112239.jpg`
- `06bbec60-20251217_110819.jpg`
- `3cca7b53-20251217_1141450.jpg`
- `53d0916d-20251217_115232.jpg`
- `fcf0a90e-20251217_114418.jpg`

### 4.3 ผลการทดลองเชิงตัวเลข
อ้างอิงผลจาก `results.csv` ที่ epoch สุดท้าย (Epoch 100)
- Precision (B) = 0.99824
- Recall (B) = 1.00000
- mAP@0.5 (B) = 0.99500
- mAP@0.5:0.95 (B) = 0.91335

ผลลัพธ์แสดงว่าโมเดลมีความสามารถสูงในการตรวจจับคลาสเป้าหมาย โดยเฉพาะที่เกณฑ์ IoU 0.5 ซึ่งได้ค่าใกล้ 1.0

### 4.4 ผลการทดลองเชิงภาพ
ไฟล์รูปสำคัญที่ใช้วิเคราะห์ผล ได้แก่
- `results.png` : แนวโน้ม loss และ metrics ตลอดการฝึก
- `confusion_matrix.png` และ `confusion_matrix_normalized.png`
- `BoxPR_curve.png`, `BoxP_curve.png`, `BoxR_curve.png`, `BoxF1_curve.png`
- `val_batch0_labels.jpg` และ `val_batch0_pred.jpg`

จากภาพรวมพบว่าเส้นโค้งและ confusion matrix สอดคล้องกับผลตัวเลขที่ดี แสดงถึงความเสถียรของโมเดลในชุด validation

### 4.5 อภิปรายผล
แม้ผลเชิงตัวเลขสูงมาก แต่ยังควรพิจารณาประเด็นต่อไปนี้
- ความสามารถในการทั่วไปกับข้อมูลนอกโดเมน (domain shift)
- ความไวต่อแสง มุมกล้อง และการบังบางส่วน
- โอกาสเกิด overfitting หากข้อมูลมีความคล้ายกันสูง

จึงควรมีการทดสอบภาคสนามเพิ่มเติมและเพิ่มความหลากหลายข้อมูลในอนาคต

## บทที่ 5 บทสรุปและข้อเสนอแนะ

### 5.1 บทสรุป
โครงการนี้สามารถพัฒนาระบบตรวจจับวัตถุแบบกำหนดคลาสเองได้ครบวงจร ตั้งแต่การเตรียมข้อมูล การฝึกโมเดล การทดสอบ ไปจนถึงการครอปภาพเพื่อนำไปใช้งานต่อ ผลการทดลองชี้ให้เห็นว่าโมเดลมีประสิทธิภาพสูงสำหรับคลาสเป้าหมาย และระบบที่พัฒนามีความพร้อมสำหรับการใช้งานจริงในระดับต้นแบบ

### 5.2 ข้อเสนอแนะ
1. เพิ่มจำนวนและความหลากหลายของข้อมูลฝึก เพื่อเพิ่มความทนทานของโมเดล
2. เพิ่มหลายคลาสใน `names` เพื่อรองรับงานที่ซับซ้อนขึ้น
3. ทดลองโมเดลขนาดอื่น (`yolov8s`, `yolov8m`) เพื่อเปรียบเทียบความแม่นยำกับเวลา
4. จัดทำชุดทดสอบภายนอกที่ไม่ซ้ำโดเมนฝึกเพื่อประเมินความสามารถการทั่วไป
5. พัฒนาระบบ deployment (เช่น API หรือแอป) เพื่อใช้งานในงานจริง

### 5.3 แนวทางพัฒนาต่อ
- เพิ่มระบบติดตามเวอร์ชันข้อมูลและโมเดล
- สร้าง dashboard สำหรับติดตามผลการทดลองหลายรอบ
- เพิ่มขั้นตอน post-processing เฉพาะโดเมนเพื่อเพิ่มความแม่นยำเชิงปฏิบัติ
